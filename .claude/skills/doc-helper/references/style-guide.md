# Documentation Style Guide

## Tone
- Professional but approachable
- Direct and concise — avoid filler words
- Second person ("you") when addressing the reader

## Formatting
- Use ATX-style headers (`#`, `##`, `###`)
- Use fenced code blocks with language identifiers (```ts, ```bash)
- Use bullet lists for unordered items, numbered lists for sequential steps
- Bold for UI elements and key terms on first use
- Inline code for file names, commands, variables, and function names

## Structure
- One idea per paragraph
- Most important information first
- Keep paragraphs to 3-4 sentences max
- Separate concepts with headers — don't write walls of text

## Code Examples
- Keep examples minimal and focused on one concept
- Use realistic but simple values (avoid `foo`/`bar` when possible)
- Show expected output when it aids understanding
